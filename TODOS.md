이번 마일스톤에서는 포함되지 않았으나, 향후 운영 및 유지보수, 그리고 더 완벽한 통계적 검증을 위해 보완해야 할 과제들입니다.

## 1. 생존자 편향(Survivorship Bias) 방지 로직 추가
- **What:** 현재 조회 시점의 거래량 상위 종목이 아닌, 과거 특정 시점(예: 1개월, 3개월 단위)을 기준으로 당시의 거래량 상위 종목 유니버스를 동적으로 재구성하는 로직.
- **Why:** 오늘 기준 우량주(거래량 많은 종목)를 기반으로 과거 2년을 테스트하면 이미 살아남아 성공한 종목만 고르게 되어 수익률이 부풀려지는 현상이 발생함.
- **Pros:** 실전과 동일한 환경의 통계적으로 유효한(Robust) 백테스트 결과를 얻을 수 있음.
- **Cons:** 과거 시점의 코스피/코스닥 전 종목 거래량 데이터를 주기적으로 받아야 하므로 쿼리가 매우 무거워지고, 종목 편출입 로직을 별도로 관리해야 함.

## 2. 정적 데이터 로컬 캐싱 및 캐시 스탬피드 대응
- **What:** 이미 종가가 확정된 과거 주가 데이터는 로컬 파일(CSV/Parquet) 또는 DB에 캐싱하고, 당일 최신 데이터만 API로 호출해서 병합(Concat)하도록 변경.
- **Why:** `app.py`에서 매시간 `ttl=3600`으로 전체 2년치 데이터를 갱신하게 되는데, 다수의 사용자가 접속할 경우 갱신 타이밍에 중복 호출(Cache Stampede)이 발생해 API 제공 측으로부터 IP 차단(Rate limit)을 당할 위험이 있음.
- **Pros:** 외부 API 호출량이 1/100 수준으로 줄어들어 속도가 극적으로 개선됨.
- **Cons:** 로컬 스토리지에 데이터를 관리하는 배치 파이프라인(스케줄러)과 파일 병합 로직이 추가로 구축되어야 함.
