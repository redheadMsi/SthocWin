## Problem Statement
기존 HTS/MTS에 의존하지 않고, 개인화된 백테스트와 실시간 자동 매매, 실시간 모니터링이 모두 가능한 독립적인 투자 환경을 구축하고자 함. 그 첫 마일스톤으로 "우량주 대상 다중 파라미터 백테스트 결과를 웹 대시보드로 비교 분석"하는 것을 목표로 함.

## What Makes This Cool
단순히 터미널에 텍스트를 출력하는 스크립트가 아니라, **Streamlit**을 이용해 웹 브라우저 상에서 동작하는 인터랙티브 대시보드를 구축. 사용자가 직접 '유니버스(지수 편입종목 vs 시총 상위)'를 고르고, 여러 개의 이평선 파라미터 조합을 한 번에 입력하여 실행(Run) 버튼을 누르면, 모든 파라미터에 대한 백테스트 결과가 리스트 형태로 출력되는 강력한 시뮬레이터가 됨.

## Constraints
- 시장: 한국 주식 시장 (코스피/코스닥)
- 개발 언어 및 도구: Python, pandas, pykrx / FinanceDataReader, Streamlit
- 무거운 기존 퀀트 프레임워크 전체 도입은 피하되, 빠른 벡터 연산이 필요한 부분은 `vectorbt`의 통계/차트 기능을 선택적으로 활용.

## Premises
1. **대상 유니버스 동적 선택:** 
   - 옵션 A: KOSPI 200 / KOSDAQ 150 지수 편입 종목
   - 옵션 B: 매일 시가총액 상위 100개 스크래핑
   사용자가 UI에서 유니버스를 선택하면 해당 풀 내에서 당일 거래대금 상위 10개 종목을 추출하고 2년 치 일봉 데이터를 가져온다. (`@st.cache_data`로 로컬 메모리에 캐싱)
2. **테스트 전략:** 이동평균선 교차 전략. 단, 고정된 값이 아니라 사용자가 여러 개의 파라미터 조합(예: [5,20], [10,60], [20,120])을 입력할 수 있다.
3. **사용자 경험 (UX):** 스크립트 실행 시 즉시 연산하지 않는다. 사용자가 옵션과 파라미터를 세팅하고 **[실행] 버튼**을 눌렀을 때만 연산이 시작되며, 진행 중임을 알리는 상태 표시(Spinner/Progress bar)가 제공되어야 한다.

## Approaches Considered
*(통합 적용)* 이전의 Approach A(지수 풀)와 Approach B(시총 스크래핑)를 모두 대시보드의 선택 옵션으로 포함하여 사용자가 시뮬레이션 시 직접 고를 수 있도록 확장함.

## Recommended Approach
**통합 시뮬레이터 대시보드 (Interactive Simulation Runner)** 채택.
단순 조회용 대시보드에서 벗어나, 사용자가 능동적으로 조건을 세팅하고 결과를 비교할 수 있는 형태의 미니 백테스트 엔진 프론트엔드를 구축.

## Success Criteria
1. 코드를 실행(`streamlit run app.py`)하면 연산 없이 세팅 UI만 먼저 표시된다.
2. UI에서 유니버스(KOSPI/KOSDAQ 풀 vs 시총 100 풀)를 선택할 수 있다.
3. UI에서 여러 개의 백테스트 파라미터(이평선 쌍)를 추가/입력할 수 있다.
4. **[실행] 버튼**을 누르면 실행 상태(진행 중 표시)가 나타나며, 선택된 유니버스의 거래대금 상위 10개 종목에 대해 입력된 모든 파라미터 조합의 백테스트가 병렬 또는 순차적으로 진행된다.
5. 연산이 완료되면, 각 파라미터 조합별 수익률 및 결과 비교 리스트가 화면에 출력된다.

## Next Steps
1. **Core 엔진 구현:** 
   - KOSPI 200 / KOSDAQ 150 티커 수집 함수
   - 시가총액 상위 100개 수집 함수
   - 위 두 가지 중 선택된 결과에서 거래대금 상위 10개를 뽑는 함수 작성 (캐싱 필수).
2. **시그널 처리:** 전달받은 파라미터 리스트를 순회하며 각각의 백테스트 수익률/승률을 계산해 결과 딕셔너리나 데이터프레임으로 반환하는 로직 작성.
3. **UI 씌우기:** `st.selectbox`(유니버스 선택), `st.text_input` 또는 동적 폼(파라미터 다중 입력), `st.button`('실행'), `st.spinner`(실행 중 상태 표시)를 활용하여 대시보드 구성.

## The Assignment
이번 주말의 단 하나 목표: **웹 UI 틀(프레임) 먼저 완성하기.** 
복잡한 백테스트 연산을 당장 완벽히 묶기 전에, 드롭다운, 파라미터 입력칸, 실행 버튼, 로딩 스피너가 있는 껍데기 UI(`app.py`)를 먼저 만들고 버튼을 눌렀을 때 가짜 결과(Dummy Data)라도 리스트로 뜨는지 확인하는 것이 첫 번째 숙제입니다.

## What I noticed about how you think
- **사용자 경험(UX)을 이해하고 있습니다:** "스크립트가 바로 실행되는 게 아니라 버튼을 눌렀을 때 실행되어야 한다", "실행 중인지 알 수 있어야 한다"는 요구사항은 전형적인 제품 제작자(Product Maker)의 시각입니다.
- **야심찬 확장(Agency):** A안과 B안 중 하나만 고르는 대신 둘 다 옵션으로 넣고 파라미터 비교까지 하겠다는 것은 범위가 커졌음을 의미합니다. 아주 멋진 방향이지만, 첫 구현 시 복잡도에 잡아먹히지 않도록 UI부터 뚫는 것이 핵심입니다.
